# Frontend Quickstart

这个 skill 用来分析整个项目，并输出分析报告、技能推荐和架构图。

适合刚接手项目、需要先把整体情况摸清时使用。

## 先运行分析脚本

开始写最终内容前，先运行：

```bash
node scripts/analyze-frontend-project.mjs <project-dir>
```

这一步会生成：

- `.frontend-quickstart/analysis.json`
- `.frontend-quickstart/analysis.md`

后面的总结、推荐和风险判断，都要优先基于这些扫描结果。

## 已有项目说明优先

在输出最终总结前，先检查目标仓库里是否已经有这类说明文件：

- `AGENT.md`
- `AGENTS.md`
- `CLAUDE.md`
- `SKILL.md`
- `.claude/`、`.codex/`、`.cursor/` 这类隐藏 agent 目录
- 其他明显重要的项目说明文件

如果这些文件存在，先读，再和仓库扫描结果一起纳入分析。

## 核心工作流

1. 先读取当前仓库结构。
2. 如果 `project-analysis.md`、`recommended-skills.md` 或之前生成的图已经存在，先删除旧文件再重新生成。
3. 先读仓库里已有的说明文件和隐藏 agent 目录里的相关上下文。
4. 运行 `node scripts/analyze-frontend-project.mjs <project-dir>`，读取 `.frontend-quickstart/analysis.json`。
5. 结合依赖清单、关键配置文件、已有说明和扫描结果，判断项目用了哪些技术，主入口在哪里，关键模块之间怎么衔接。
6. 使用内置的 `find-skills` 推荐相关 skills，并在写入 `recommended-skills.md` 前确认每个 skill 的精确 install id 和 GitHub 来源一致。
7. 使用内置的 `receiving-code-review` 补充风险判断，找出更值得优先看的模块。
8. 默认从内部分析结果里排在前 3、最该先看的模块里，拿第 1 名用内置的 `performance-optimizer` 做一次性能分析。对有意义的修改建议做详细展开，其余补文件路径、行号和简短原因。
9. 用内置的 `risk-hotspot-mermaid` 补一张 Mermaid 关系图，把最该先看的模块和它们之间的关系画出来，再把图和对应的文字原因一起写进去。
10. 如果最终要交付中文内容，用内置的 `project-translate` 把 `project-analysis.md`、`recommended-skills.md` 里的英文草稿段落收成自然中文，再写入最终文件。
11. 生成 `project-analysis.md`、`recommended-skills.md` 和架构图。
12. 结束前运行 `node scripts/validate-frontend-output.mjs <project-dir>`。

不要在项目目录或 `.frontend-quickstart/` 里写任何 `gsd` 上下文文件；`gsd` 只用于线程内整理，不进入分析结果目录。

## 内置 Skills

这个 skill 内置了：

- `skills/find-skills/`：用于推荐相关 skills
- `skills/receiving-code-review/`：用于补充风险判断
- `skills/performance-optimizer/`：用于继续分析具体模块的性能问题和修改方向
- `skills/risk-hotspot-mermaid/`：用于生成 Mermaid 关系图，说明哪些模块该先看
- `skills/project-translate/`：用于把最终报告和推荐清单收成自然中文
- `skills/frontend-diagram/`：用于生成架构图

## 推荐 Skills

只推荐外部公开、可安装、并且对当前项目最贴切的 skill。能直接对应当前技术栈时优先直接对应；如果没有，就退到同一层里最接近、最有帮助的 skill。先补齐基础技术栈，再补充子系统相关的 skill。

对于每个推荐项：

- 包含精确可安装的 skill id，格式为 `owner/repo@skill`
- 包含一句自然的描述
- 必须包含 GitHub 仓库地址，并使用短标签链接
- 包含一条与项目当前技术选择直接相关的简短理由

`recommended-skills.md` 开头直接进入推荐结果。可以从标题后直接接表格；如果确实需要过渡句，也只保留一句很短的引导句。

每个推荐项都必须能对应到同一个公开 skill：install id、来源仓库和 GitHub 链接要一致；拿不准就不要写。但推荐本身不需要死卡成“依赖名必须一字不差地对应到 skill 名”。

## 项目分析报告

报告至少应包含：

- 项目主要用到的技术
- 如果存在的话，仓库里已有的项目说明、隐藏 agent 目录上下文，以及 agent/skill 约定
- 建议先按什么顺序读
- 建议先看的几个模块
- 对最该先看的前三个模块里第 1 名的性能分析结果
- 后面再看哪里

如果最终文件要用中文，`project-analysis.md` 和 `recommended-skills.md` 在写入前要统一收成中文表达，不要保留整段英文草稿。

内容要简洁、结构化，并且直接基于仓库事实。

`project-analysis.md` 使用固定小节结构。默认按下面这组章节来写：

- 先按这个顺序读
- 建议先看的几个模块
- 性能分析
- 后面再看哪里

每个小节按固定槽位输出：

- 标题下摘要
  只写 2 到 3 句。第 1 句说明项目主要是什么；第 2 句点出最值得优先注意的主系统；第 3 句只在确实需要时补一句已有项目说明与代码是否一致。
  摘要里的判断句必须立刻落到具体对象：同一句或紧接着下一句就要点出明确的路径、目录、模块或文件，不能先写一层抽象结论，再把代码名拖到后面。
- `先按这个顺序读`
  只写 1 条主阅读顺序，然后补一小段解释，说明为什么先按这个顺序读。
- `建议先看的几个模块`
  只写排在前面的几个模块，每个模块后面只接原因句，不展开成长段。
- `性能分析`
  固定写 5 段：
  1. 解释
  2. 运行时间分析
  3. 可以先改的地方
  4. 影响力与复杂度表
  5. 展开的修改建议的详细说明、改之前先注意和改法示意
     如果有意义的修改项少于 5 个，就把这些项全部展开；如果多于 5 个，就只展开前 5 个，剩下的直接写成带路径、行号和简短理由的普通列表。
     这一段必须把第 1、第 2、第 3 处分别写全，不能只展开第 1 处。
- `后面再看哪里`
  只写固定阅读顺序，不再额外开新分析分支。

默认先按这 5 个章节写。只有遇到单独写一段才讲得清的复杂部分，再加新标题。

`project-analysis.md` 要写成项目分析，不要按目录从上到下介绍。

先写代码里能直接确认的事实，再写由这些事实能推出的判断，不要把判断写得比代码更重。

开头摘要和首读说明都按这个顺序写：先点出代码里直接能看到的文件、目录、调用关系或状态来源，再补一句这意味着什么。不要先写“真正的复杂度集中在……”“真正的控制交接顺序是……”这类总括句，再回头补代码名。

章节结构贴着当前项目来写；只有确实需要时，才单独展开仓库结构、包边界或其他复杂区域。

重点放在最值得先读的顺序、模块和风险上，不要写成评论、印象或空泛概括。

写项目重点时，优先直接用仓库里已经存在的路径、目录名、模块名和文件名，不要自己补新的概括叫法。

不要只停留在目录介绍。报告里还必须给出判断：

- 建议先看哪几个模块
- 更可能出现哪一类问题
- 这个判断为什么能从当前代码结构里推出
- 如果已有项目说明存在，这些说明如何改变或补充你的判断

分析运行时问题时，要按当前项目自己的技术栈来写，只讨论当前生态里确实成立的运行时边界、状态流、生命周期和性能成本。相关判断要贴着代码事实写，不要自己补仓库里没有的概念。

性能分析里的说明句，先写“现在这段代码怎么做”，再写“为什么这里会慢、会重、会重复，或者容易出问题”，最后再写“改的时候什么不能动”。每一句都要能对应到当前代码里看得见的动作、数据路径或更新路径。

写文件、store、service 和界面之间的关系时，直接写谁在读什么、谁在改什么、页面上哪一部分会跟着变。

写临时存起来的数据时，直接写这些数据现在放在哪里、什么时候再拿出来用，不要只丢一个简称。

如果某个复杂系统被判断为核心架构里的主要复杂度来源，就继续写清它的具体组成：

- 入口文件或协调文件在哪里
- 主要状态源或数据源有哪些
- 哪些职责缠在一起
- 更可能出现什么失效方式或维护风险

不要只下结论，要把问题为什么容易出在这里写清楚。

默认对最该先看的前三个模块里排第 1 的那个做一次 `performance-optimizer` 性能分析，并把结果写进 `project-analysis.md`。这里写的是局部优化方向，不是完整替换稿。

必须补一张 Mermaid 关系图。默认围绕内部分析结果里最值得先看的那个模块来画，用 `flowchart` 按页面、状态、数据与连接分组；图下面再补简短说明。代码关系很明确时，可以在箭头旁边补一两个简短动词。

Mermaid 图里的箭头文字不能比正文证据更重。只有正文已经明确写出“谁读什么”“谁改什么”“谁和谁同步”时，箭头旁边才补 `reads`、`writes`、`syncs` 这类短词；如果正文还没把关系说实，就只画连线，不写标签。

颜色图例固定为：

- 红色：先看这里
- 橙色：接着看这里
- 灰蓝色：相关模块

如果一张图放不下，就拆成两张或多张 Mermaid 图，不要硬塞进一张。

报告里至少给出一条优先阅读路径，不要只罗列文件。

优先指出同时牵涉路由、状态、请求或 UI 的模块，这些地方通常更值得先看。

可以参考 `.frontend-quickstart/analysis.json`，但这些字段只用于内部排序、选重点和交叉验证判断，不要把内部统计对象直接写进最终报告。

最终报告里不要交代内部分析过程本身。不要写“分析脚本把它排进 top 3 风险模块”“内部分析结果显示它是第 1 名”这类句子；直接写结论和代码理由。

优先覆盖这些问题：

- 这个项目主要是什么
- 建议先按什么顺序读
- 哪几个模块最值得先看，为什么
- 如果刚接手，先看哪里最有效
- 复杂度主要集中在哪些地方

## 默认规则

- 当 README 与代码或配置不一致时，以代码和配置为准。
- 在可能的情况下，优先推荐有稳定 GitHub 仓库的公开 skills。
- 推荐必须基于项目当前实际使用的技术，而不是猜测。

## 输出

最终交付：

- `project-analysis.md`
- `recommended-skills.md`
- 架构图
